# SLCTS PostgreSQL Database Layer

**Smart Logistics & Cargo Transit System (SLCTS)** — production relational schema for users, fleet, bookings, and financial ledger.

DDL source files live in [`../backend/sql/`](../backend/sql/). This document describes the data model, initialization workflow, and integrity rules enforced at the database level.

---

## 1. Overview

The SLCTS database is a **PostgreSQL 14+** relational store designed for a multi-role logistics platform (customers, drivers, admins). The schema prioritizes three production concerns:

### UUID identifiers

Every primary key (`users`, `drivers`, `vehicles`, `bookings`, `transaction_ledger`) is a **`UUID`** generated by `gen_random_uuid()`. UUIDs avoid sequential ID leakage, simplify distributed service references, and align with JWT claims (`userId`, `driverId`) in the Node.js API layer.

### Strict decimal calculations for UGX finances

All monetary fields use **`NUMERIC(12,2)`** — never `FLOAT` or `REAL` — so Ugandan Shilling (UGX) amounts are stored and compared without floating-point drift.

| Column | Table | Behavior |
|--------|-------|----------|
| `final_fare_ugx` | `bookings` | Trip fare; must be `> 0` when set |
| `total_amount_gross_ugx` | `transaction_ledger` | Gross trip payment; must be `> 0` |
| `platform_commission_ugx` | `transaction_ledger` | **Generated column**: `ROUND(gross × 0.15, 2)` |
| `driver_net_payout_ugx` | `transaction_ledger` | **Generated column**: `ROUND(gross × 0.85, 2)` |

Commission and driver payout are computed in the database, not application code, guaranteeing a consistent **15% / 85%** split on every ledger row.

### Physical data constraints

Cargo and vehicle capacity use **`NUMERIC(6,2)`** for payload and weight bounds (tonnes). Distance uses the same precision (kilometres). PostgreSQL `ENUM` types lock domain values (trip status, cargo type, escrow state) so invalid states cannot be inserted even if application validation is bypassed.

Foreign keys use deliberate delete semantics:

- **`ON DELETE CASCADE`** — removing a `user` removes linked `drivers`; removing a `driver` removes their `vehicles`.
- **`ON DELETE RESTRICT`** — `bookings.customer_id` and `transaction_ledger.booking_id` block accidental deletion of financial records.
- **`ON DELETE SET NULL`** — unassigning a `driver` from a `booking` when the driver row is removed.

`updated_at` columns are maintained automatically via the shared `trigger_set_updated_at()` trigger on all core tables.

## 2. Entity-Relationship Quick Map

```
users (1) ──────< (0..1) drivers (1) ──────< (1..*) vehicles
  │                      │
  │                      └── is_online, verification_status, permit_number
  │
  └──< (1..*) bookings >── (0..1) drivers
              │
              └── (1) transaction_ledger
```

### `users`

Platform accounts for customers, drivers, and admins.

| Column | Notes |
|--------|-------|
| `id` | UUID PK |
| `name`, `phone_number`, `email` | `phone_number` and `email` are **UNIQUE** |
| `password_hash` | bcrypt hash (VARCHAR 255) |
| `role` | `CUSTOMER` \| `DRIVER` \| `ADMIN` |
| `is_active` | Soft-enable flag |

### `drivers`

Operational profile linked 1:1 to a `users` row with `role = DRIVER`.

| Column | Notes |
|--------|-------|
| `id` | UUID PK |
| `user_id` | FK → `users.id` (CASCADE) |
| `permit_number` | **UNIQUE** regulatory identifier |
| `verification_status` | `PENDING` \| `APPROVED` \| `REJECTED` |
| `is_online` | Availability for dispatch matching |
| `gateway_subaccount_id` | Payment gateway integration (optional) |

### `vehicles`

One driver may operate one or more registered vehicles; each row defines tier and weight capacity.

| Column | Notes |
|--------|-------|
| `id` | UUID PK |
| `driver_id` | FK → `drivers.id` (CASCADE) |
| `license_plate` | **UNIQUE** |
| `tier_class` | `SMALL` \| `MEDIUM` \| `LARGE` |
| `min_weight_tonnes`, `max_weight_tonnes` | Capacity bounds for cargo matching |

### `bookings`

A cargo trip from pickup to drop-off, optionally assigned to a driver.

| Column | Notes |
|--------|-------|
| `id` | UUID PK |
| `customer_id` | FK → `users.id` (RESTRICT) |
| `driver_id` | FK → `drivers.id` (SET NULL) — nullable until accepted |
| `pickup_address`, `dropoff_address` | Free-text locations |
| `cargo_type` | `BULK`, `GENERAL`, `HEAVY_MACHINERY`, `PERISHABLE`, `HIGH_VALUE` |
| `payload_weight_tonnes` | Must be `> 0` |
| `calculated_distance_km`, `final_fare_ugx` | Optional until priced |
| `trip_status` | Lifecycle: `REQUESTED` → `ACCEPTED` → `EN_ROUTE` → `ARRIVED` → `COMPLETED` / `CANCELLED` |
| `requested_at`, `completed_at` | Trip timing |

### `transaction_ledger`

One financial record per completed booking (1:1 via `booking_id` UNIQUE).

| Column | Notes |
|--------|-------|
| `id` | UUID PK |
| `booking_id` | FK → `bookings.id` (RESTRICT), **UNIQUE** |
| `transaction_reference` | External payment ref, **UNIQUE** |
| `total_amount_gross_ugx` | Gross fare collected |
| `platform_commission_ugx` | Auto: 15% of gross |
| `driver_net_payout_ugx` | Auto: 85% of gross |
| `escrow_state` | `HELD` \| `RELEASED` \| `REFUNDED` |
| `released_at` | Set when admin releases escrow to driver |

## 3. Local Initialization Guide

### Prerequisites

- PostgreSQL 14+ with `psql` on your `PATH`
- `gen_random_uuid()` available (included in `pgcrypto` on older versions; built-in on PostgreSQL 13+)

### Option A — Automated (Windows PowerShell)

From the repository root:

```powershell
cd backend
.\setup-db.ps1
```

This script creates the `slcts` database (if missing), applies `backend/sql/schema.sql`, and writes `backend/.env` with `DATABASE_URL`.

### Option B — Manual `psql` (cross-platform)

**Step 1 — Set connection credentials**

```bash
export PGHOST=localhost
export PGPORT=5432
export PGUSER=postgres
export PGPASSWORD=your_password
```

**Step 2 — Create the database**

```bash
psql -U "$PGUSER" -h "$PGHOST" -p "$PGPORT" -tc \
  "SELECT 1 FROM pg_database WHERE datname = 'slcts'" | grep -q 1 \
  || psql -U "$PGUSER" -h "$PGHOST" -p "$PGPORT" -c "CREATE DATABASE slcts;"
```

**Step 3 — Load the canonical DDL**

```bash
psql "postgresql://$PGUSER:$PGPASSWORD@$PGHOST:$PGPORT/slcts" \
  -f backend/sql/schema.sql
```

The schema file is **idempotent** (`IF NOT EXISTS`, `CREATE OR REPLACE`) — safe to re-run on an existing database.

**Step 4 — Apply migrations (if upgrading an older database)**

```bash
psql "postgresql://$PGUSER:$PGPASSWORD@$PGHOST:$PGPORT/slcts" \
  -f backend/sql/002_auth_password_hash.sql
```

> **Note:** `schema.sql` already includes `password_hash` on `users`. The migration is only needed for databases created before that column was added.

**Step 5 — Confirm table creation**

```bash
psql "postgresql://$PGUSER:$PGPASSWORD@$PGHOST:$PGPORT/slcts" -c "\dt"
```

Expected output includes:

```
 public | bookings           | table | ...
 public | drivers            | table | ...
 public | transaction_ledger | table | ...
 public | users              | table | ...
 public | vehicles           | table | ...
```

**Step 6 — Verify enums and generated columns**

```bash
psql "postgresql://$PGUSER:$PGPASSWORD@$PGHOST:$PGPORT/slcts" -c \
  "SELECT column_name, data_type, is_generated FROM information_schema.columns
   WHERE table_name = 'transaction_ledger' AND column_name LIKE '%ugx%';"
```

**Step 7 — Seed the admin user (application layer)**

```bash
cd backend
npm run seed:admin
```

Copy `backend/.env.example` to `backend/.env`, set `DATABASE_URL` and `JWT_SECRET`, then start the API with `npm run dev`.

## 4. Data Integrity Guardrails

PostgreSQL `CHECK` constraints enforce business rules that must hold regardless of application bugs or ad-hoc SQL.

### Vehicle weight capacity (`vehicles`)

| Constraint | Rule | Purpose |
|------------|------|---------|
| `min_weight_tonnes > 0` | Minimum capacity is positive | Prevents zero or negative tier bounds |
| `max_weight_tonnes > 0` | Maximum capacity is positive | Prevents invalid upper bounds |
| `chk_weight_range` | `max_weight_tonnes > min_weight_tonnes` | Ensures a meaningful capacity band for dispatch matching |

Dispatch logic joins `bookings.payload_weight_tonnes` against these bounds to offer trips only to suitably tiered vehicles.

### Booking physical and financial fields (`bookings`)

| Constraint | Rule | Purpose |
|------------|------|---------|
| `payload_weight_tonnes > 0` | Cargo weight must be positive | Blocks empty or negative shipments |
| `calculated_distance_km > 0` | Distance must be positive when set | Prevents invalid route metrics |
| `final_fare_ugx > 0` | Fare must be positive when set | Blocks zero or negative trip pricing |

### Ledger currency (`transaction_ledger`)

| Constraint | Rule | Purpose |
|------------|------|---------|
| `total_amount_gross_ugx > 0` | Gross payment must be positive | No zero-value or negative transactions |
| `chk_commission_and_payout` | `commission + payout = gross` (rounded) | Guarantees the 15%/85% split sums exactly to gross UGX |

Generated columns (`platform_commission_ugx`, `driver_net_payout_ugx`) are **STORED** — values are computed at insert/update time and indexed like regular columns, so reporting queries never depend on application-side math.

### Referential integrity summary

| Relationship | On delete | Rationale |
|--------------|-----------|-----------|
| `drivers.user_id` → `users` | CASCADE | Driver profile is owned by the user account |
| `vehicles.driver_id` → `drivers` | CASCADE | Vehicles belong to the driver |
| `bookings.customer_id` → `users` | RESTRICT | Cannot delete a customer with active bookings |
| `bookings.driver_id` → `drivers` | SET NULL | Preserve booking history if driver is removed |
| `transaction_ledger.booking_id` → `bookings` | RESTRICT | Financial records are immutable anchors |
